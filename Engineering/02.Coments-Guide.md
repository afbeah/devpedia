# Comments Guide

Comentários são anotações inseridas no código com o objetivo de explicar regras de negócio, decisões técnicas ou comportamentos que não são imediatamente óbvios.

Seu principal objetivo é facilitar a compreensão e manutenção do software.

Em outras palavras:

> Comentários devem complementar o código, não substituir código mal escrito.

## Por que utilizar?

Durante o desenvolvimento existem situações onde apenas o código não é suficiente para transmitir toda a intenção da implementação.

Exemplos:

* regras de negócio complexas;
* decisões arquiteturais;
* integrações externas;
* limitações técnicas;
* comportamentos temporários.

Nesses casos, comentários podem ajudar a documentar o contexto.

## O problema dos comentários

Muitos desenvolvedores acreditam que comentar mais significa produzir um código melhor.

Exemplo:

```java
// Soma dois números
return a + b;
```

O comentário não adiciona informação relevante.

O próprio código já explica o que está acontecendo.

## O que comentar?

### Regras de negócio

```java
// Clientes VIP possuem 20% de desconto por exigência contratual
```

### Decisões técnicas

```java
// Utilizamos cache local para reduzir chamadas à API externa
```

### Limitações conhecidas

```java
// Limitação temporária até a migração para o novo provedor
```

### Integrações complexas

```java
// Endpoint legado que retorna status invertido
```

## O que evitar?

### Comentários óbvios

Ruim:

```java
// Incrementa contador
counter++;
```

### Código comentado

Ruim:

```java
// calculateOldPrice();
```

O Git já preserva histórico de alterações.

### Comentários desatualizados

Ruim:

```java
// Calcula desconto de 10%
```

Mas o código já calcula 20%.

Comentários incorretos são piores do que não ter comentários.

## Tipos de Comentários

### Comentário de Linha

Utilizado para observações rápidas.

```java
// Aplicar regra de desconto para clientes premium
```

### Comentário de Bloco

Utilizado para explicações maiores.

```java
/*
Processamento realizado em lote
para reduzir carga no banco.
*/
```

### Comentário de Documentação

Utilizado para gerar documentação automática.

Exemplo em Java:

```java
/**
 * Retorna o usuário pelo identificador.
 */
```

Exemplo em Go:

```go
// GetUser retorna um usuário pelo ID informado.
func GetUser(id string) {}
```

## Comentários por Linguagem

### C / C++ / Java / JavaScript

Linha única:

```java
// comentário
```

Bloco:

```java
/* comentário */
```

### Go

```go
// comentário
```

### Python

Linha única:

```python
# comentário
```

### HTML

```html
<!-- comentário -->
```

### CSS

```css
/* comentário */
```

## Comentários e Clean Code

Existe uma forte relação entre os dois conceitos.

Clean Code defende:

```txt
Código legível
↓
Menos necessidade de comentários
```

O ideal é que nomes de variáveis, funções e classes expliquem grande parte da intenção do código.

Exemplo:

Ruim:

```java
// Soma valores do carrinho
calculate();
```

Melhor:

```java
calculateCartTotal();
```

Nesse caso o comentário torna-se desnecessário.

## Boas Práticas

* Comente o "porquê", não apenas o "o quê";
* Prefira código legível antes de adicionar comentários;
* Mantenha comentários atualizados;
* Documente regras de negócio importantes;
* Utilize comentários de documentação em APIs públicas;
* Evite código comentado.

## Quando utilizar?

Comentários costumam ser úteis para:

* regras de negócio;
* integrações externas;
* decisões arquiteturais;
* documentação pública;
* comportamentos não intuitivos.

## Quando evitar?

Comentários podem ser desnecessários quando:

* apenas repetem o código;
* explicam operações simples;
* escondem problemas de legibilidade.

## Relação com outros conceitos

Fluxo comum:

```txt
Clean Code
↓
Comments Guide
↓
KISS
↓
DRY
↓
SOLID
```

Todos esses conceitos contribuem para a criação de software mais legível e fácil de manter.

## Resumo

Comentários são ferramentas utilizadas para complementar a compreensão do código.

Quando bem utilizados, ajudam a documentar regras de negócio, decisões técnicas e comportamentos complexos. Quando utilizados em excesso ou de forma incorreta, podem gerar ruído e dificultar a manutenção do software.
