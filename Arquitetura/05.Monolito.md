# Monolito

Monolito é um estilo arquitetural onde toda a aplicação é desenvolvida, implantada e executada como uma única unidade.

Nesse modelo, todas as funcionalidades do sistema fazem parte da mesma aplicação.

## O que é um Monolito?

Imagine um sistema contendo:

* usuários;
* pedidos;
* pagamentos;
* produtos;
* notificações.

Em uma arquitetura monolítica, todos esses módulos pertencem à mesma aplicação.

Exemplo:

```txt id="a7s4mb"
Aplicação
├── Usuários
├── Pedidos
├── Produtos
├── Pagamentos
└── Notificações
```

Tudo é executado em um único processo.

## Como funciona?

Quando o sistema é iniciado:

```txt id="wbec6z"
Aplicação
↓
Todos os módulos carregados
↓
Sistema disponível
```

Quando ocorre um deploy:

```txt id="r7v8sv"
Aplicação inteira
↓
Deploy
```

Toda a aplicação é publicada em conjunto.

## Exemplo prático

Imagine um e-commerce.

Estrutura:

```txt id="omcd0o"
src/
├── users
├── products
├── orders
├── payments
└── notifications
```

Embora existam vários módulos, todos pertencem à mesma aplicação.

## Benefícios

### Simplicidade

O desenvolvimento inicial costuma ser mais simples.

```txt id="yz0e3f"
Uma aplicação

Um deploy

Um ambiente
```

### Facilidade de desenvolvimento

Em projetos pequenos e médios, a comunicação entre módulos é direta.

Exemplo:

```txt id="0wml3w"
Pedido
↓
Pagamento
```

Sem necessidade de APIs ou mensageria.

### Menor complexidade operacional

Não é necessário gerenciar:

* múltiplos serviços;
* comunicação distribuída;
* observabilidade complexa.

## Desafios

### Crescimento da aplicação

Com o tempo, o sistema pode ficar muito grande.

Exemplo:

```txt id="5gppkh"
500 mil linhas de código
```

ou

```txt id="e9dl5e"
Dezenas de equipes
```

### Deploy único

Uma pequena alteração exige a publicação da aplicação inteira.

```txt id="t41b7s"
Mudança em Notificações
↓
Deploy completo
```

### Escalabilidade limitada

Nem sempre é possível escalar apenas uma parte do sistema.

Exemplo:

```txt id="t0y8p3"
Pagamentos precisa crescer
↓
Aplicação inteira é escalada
```

## Monolito Modular

Nem todo monolito é desorganizado.

Uma prática comum é dividir a aplicação em módulos.

Exemplo:

```txt id="z8m3dt"
Users

Orders

Payments

Products
```

Mesmo sendo uma única aplicação, existe separação de responsabilidades.

Esse modelo é conhecido como Monolito Modular.

## Monolito x Microsserviços

### Monolito

```txt id="nvdpgk"
Uma aplicação

Um deploy

Um banco (geralmente)
```

### Microsserviços

```txt id="gdfz7o"
Várias aplicações

Deploy independente

Comunicação entre serviços
```

## Quando utilizar?

Monolitos costumam ser uma boa escolha para:

* sistemas pequenos;
* MVPs;
* startups;
* equipes reduzidas;
* produtos em estágio inicial.

## Quando reavaliar?

O modelo pode começar a apresentar limitações quando existem:

* muitas equipes;
* alto volume de acessos;
* necessidade de deploys independentes;
* crescimento excessivo da base de código.

## Monolito e Arquitetura

Monolito é um estilo arquitetural.

Isso significa que ele define como os componentes do sistema serão organizados e implantados.

Importante:

> Monolito não significa código ruim.

É possível ter:

```txt id="0qh5y2"
Monolito
+
SOLID
+
Clean Code
+
Clean Architecture
```

e manter um sistema extremamente organizado.

## Exemplos famosos

Muitas empresas começaram utilizando monólitos.

Exemplos conhecidos incluem versões iniciais de:

* GitHub;
* Shopify;
* Basecamp.

Antes de adotarem arquiteturas distribuídas, cresceram utilizando aplicações monolíticas.

## Resumo

Monolito é um estilo arquitetural onde todas as funcionalidades da aplicação são executadas como uma única unidade.

Ele oferece simplicidade de desenvolvimento e operação, sendo uma excelente escolha para muitos projetos. Apesar das limitações em cenários muito grandes, continua sendo uma solução válida e amplamente utilizada na indústria de software.
