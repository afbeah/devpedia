# Domain-Driven Design (DDD)

Domain-Driven Design (DDD) é uma abordagem para desenvolvimento de software focada no domínio do negócio.

Foi proposta por Eric Evans no livro *Domain-Driven Design: Tackling Complexity in the Heart of Software*.

O principal objetivo do DDD é aproximar o software das regras e necessidades reais do negócio.

Em outras palavras:

> O software deve refletir a forma como o negócio funciona.

## O que é Domínio?

Domínio é a área de conhecimento que o sistema resolve.

Exemplos:

### E-commerce

```txt id="a6z71g"
Cliente
Produto
Pedido
Pagamento
Entrega
```

### Banco

```txt id="d4s91h"
Conta
Transferência
Saldo
Pagamento
```

### Hospital

```txt id="w6z23f"
Paciente
Consulta
Médico
Exame
```

Esses conceitos formam o domínio do sistema.

## O problema que o DDD resolve

Em muitos projetos, o código acaba sendo organizado apenas por aspectos técnicos.

Exemplo:

```txt id="x8p4nb"
Controller
Service
Repository
Model
```

Embora essa estrutura funcione, ela nem sempre representa o negócio.

O DDD propõe organizar o sistema de forma que os conceitos do negócio sejam o foco principal.

## Linguagem Ubíqua (Ubiquitous Language)

Um dos conceitos mais importantes do DDD.

A equipe técnica e a equipe de negócio devem utilizar os mesmos termos.

Exemplo:

Se o negócio fala:

```txt id="e3f7wa"
Pedido
```

O sistema deve utilizar:

```java id="p6d9sz"
Order
```

e não:

```java id="v2j4qh"
TransactionData
```

A linguagem deve ser consistente em:

* código;
* documentação;
* reuniões;
* modelagem.

## Entidades (Entities)

Representam objetos que possuem identidade própria.

Exemplo:

```txt id="d9f4cw"
Cliente
```

Mesmo que os dados mudem:

```txt id="j8r2nv"
Nome

Telefone

Endereço
```

continua sendo o mesmo cliente.

Exemplo:

```java id="r5x7pf"
Customer
```

## Objetos de Valor (Value Objects)

Representam conceitos sem identidade própria.

Exemplo:

```txt id="n4w8ep"
CPF

Endereço

Email
```

Se dois objetos possuem exatamente os mesmos valores, eles são considerados iguais.

Exemplo:

```java id="s9t3hz"
Address
```

## Agregados (Aggregates)

Agrupam entidades relacionadas.

Exemplo:

```txt id="4m7dju"
Pedido
├── Itens
├── Pagamento
└── Cliente
```

O agregado define limites e regras de consistência.

## Repositórios (Repositories)

Responsáveis por acessar e persistir agregados.

Exemplo:

```java id="u3e5lm"
OrderRepository
```

Eles escondem detalhes de armazenamento.

## Serviços de Domínio

Utilizados quando uma regra de negócio não pertence claramente a uma única entidade.

Exemplo:

```txt id="g8q4ya"
Cálculo de frete

Processamento de pagamento
```

## Bounded Context

Um dos conceitos mais importantes do DDD.

Um mesmo termo pode possuir significados diferentes dependendo do contexto.

Exemplo:

```txt id="7u4fpm"
Pedido
```

No contexto financeiro:

```txt id="c6n8xb"
Cobrança
```

No contexto logístico:

```txt id="w2m5rk"
Entrega
```

Cada contexto possui suas próprias regras.

## DDD e Microsserviços

DDD é frequentemente utilizado para definir limites entre microsserviços.

Exemplo:

```txt id="z7y3nl"
Customer Context
↓
Customer Service

Order Context
↓
Order Service

Payment Context
↓
Payment Service
```

O domínio ajuda a determinar onde um serviço começa e termina.

## DDD e Modelagem

Modelagem e DDD possuem forte relação.

Fluxo:

```txt id="j3v9tb"
Negócio
↓
Modelagem
↓
DDD
↓
Implementação
```

O DDD utiliza a modelagem para representar corretamente o domínio.

## Benefícios

* Melhor alinhamento com o negócio;
* Código mais expressivo;
* Redução de ambiguidades;
* Facilidade de evolução;
* Melhor definição de responsabilidades.

## Desafios

* Curva de aprendizado;
* Necessidade de conhecer profundamente o negócio;
* Complexidade em projetos pequenos.

## Quando utilizar?

DDD costuma trazer mais benefícios em:

* sistemas corporativos;
* domínios complexos;
* microsserviços;
* aplicações de longo prazo.

## Quando evitar?

Projetos simples podem não justificar toda a complexidade conceitual.

Exemplo:

```txt id="c4z8lw"
Landing Pages

Sites institucionais

Protótipos
```

## DDD NÃO é

DDD não é:

* uma linguagem;
* um framework;
* uma arquitetura;
* uma ferramenta.

DDD é uma abordagem para modelar software com foco no negócio.

## Exemplo prático

Imagine um e-commerce.

Sem DDD:

```txt id="t7r4ev"
Users

Products

Orders
```

Com DDD:

```txt id="v9y2ah"
Customer Context

Catalog Context

Order Context

Payment Context
```

Os limites do negócio tornam-se mais claros.

## Resumo

Domain-Driven Design (DDD) é uma abordagem para desenvolvimento de software que coloca o domínio do negócio no centro da aplicação.

Seu objetivo é criar sistemas que representem fielmente as regras e processos do negócio, utilizando uma linguagem comum entre especialistas e desenvolvedores para reduzir complexidade e melhorar a evolução do software.
