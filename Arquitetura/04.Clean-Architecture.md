# Clean Architecture

Clean Architecture é um modelo arquitetural proposto por Robert C. Martin (Uncle Bob) com o objetivo de criar sistemas mais organizados, testáveis e independentes de tecnologias externas.

Seu principal objetivo é separar as regras de negócio da infraestrutura.

Em outras palavras:

> O negócio deve ser o centro da aplicação.

## O problema que a Clean Architecture tenta resolver

Em muitos sistemas, a lógica de negócio acaba misturada com:

* banco de dados;
* frameworks;
* APIs externas;
* interfaces gráficas;
* infraestrutura.

Exemplo:

```txt id="9eh7nn"
Regra de negócio
+
Banco de Dados
+
Framework
+
API
```

Com o crescimento da aplicação, a manutenção torna-se mais difícil.

A Clean Architecture busca reduzir esse acoplamento.

## Princípio fundamental

A regra mais importante da Clean Architecture é:

> Dependências devem apontar para dentro.

Ou seja:

```txt id="qjlwmw"
Infraestrutura
↓
Aplicação
↓
Domínio
```

O domínio não deve conhecer detalhes da infraestrutura.

## Estrutura em camadas

A representação clássica utiliza círculos concêntricos.

```txt id="f85cxn"
Frameworks e Drivers
↓
Interface Adapters
↓
Use Cases
↓
Entities
```

Quanto mais próximo do centro, mais importante é a regra de negócio.

## Entities

Representam as regras de negócio centrais da aplicação.

Exemplo:

```txt id="h6c9zd"
Usuário
Pedido
Produto
Pagamento
```

Essas regras devem ser independentes de qualquer tecnologia.

## Use Cases

Representam ações que o sistema executa.

Exemplos:

```txt id="4g5d0h"
Criar Pedido

Cadastrar Usuário

Processar Pagamento
```

Os casos de uso utilizam as entidades para resolver problemas do negócio.

## Interface Adapters

Adaptam informações entre as camadas.

Exemplos:

```txt id="9h92gi"
Controllers

Presenters

Repositories
```

Funcionam como intermediários.

## Frameworks e Drivers

Camada mais externa.

Exemplos:

```txt id="f4e2yv"
Spring Boot

PostgreSQL

MongoDB

RabbitMQ

Kafka
```

Esses elementos podem ser substituídos sem alterar a lógica de negócio.

## Exemplo visual

```txt id="tw2wx8"
Controller
↓
Use Case
↓
Entity
```

A regra de negócio permanece protegida das tecnologias externas.

## Exemplo prático

Imagine um caso de uso:

```txt id="8zw2jz"
Criar Pedido
```

Fluxo:

```txt id="j2v9ld"
Controller
↓
CreateOrderUseCase
↓
Order
↓
Repository
↓
Banco
```

O Use Case não precisa saber se o banco é:

* PostgreSQL;
* MySQL;
* MongoDB.

Ele conhece apenas uma abstração.

## Clean Architecture e Repository

O padrão Repository é muito utilizado nessa arquitetura.

Exemplo:

```go id="b63l0x"
type UserRepository interface {
    Save(user User) error
}
```

Implementação:

```go id="zn90a8"
type PostgresUserRepository struct {}
```

O caso de uso depende da interface.

Não da implementação concreta.

## Benefícios

* Baixo acoplamento;
* Alta testabilidade;
* Facilidade de manutenção;
* Independência de frameworks;
* Facilidade de evolução.

## Desafios

* Maior quantidade de arquivos;
* Curva de aprendizado;
* Complexidade inicial;
* Possibilidade de excesso de abstrações.

## Clean Architecture x MVC

### MVC

Foco na separação entre:

```txt id="0l6ag8"
Model
View
Controller
```

### Clean Architecture

Foco na separação entre:

```txt id="g33efm"
Domínio
Casos de Uso
Infraestrutura
```

MVC organiza a aplicação.

Clean Architecture protege as regras de negócio.

## Clean Architecture x Clean Code

Esses conceitos são diferentes.

### Clean Code

Preocupa-se com:

```txt id="my74pz"
Como escrever código
```

### Clean Architecture

Preocupa-se com:

```txt id="vlvwd4"
Como organizar o sistema
```

Os dois conceitos se complementam.

## Quando utilizar?

Clean Architecture costuma ser utilizada em:

* sistemas corporativos;
* APIs;
* microsserviços;
* aplicações de médio e grande porte.

## Quando evitar?

Projetos muito pequenos podem não se beneficiar da complexidade adicional.

Exemplo:

```txt id="1p0f7f"
Landing Pages

Protótipos

Aplicações simples
```

Nesses casos uma arquitetura mais simples pode ser suficiente.

## Resumo

Clean Architecture é um modelo arquitetural que organiza sistemas em camadas independentes, colocando as regras de negócio no centro da aplicação.

Seu objetivo é reduzir acoplamento, aumentar testabilidade e permitir que a aplicação evolua sem depender diretamente de frameworks, bancos de dados ou tecnologias específicas.
